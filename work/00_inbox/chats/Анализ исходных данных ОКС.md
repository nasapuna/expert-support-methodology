# Анализ исходных данных как SourceSignalIntake

Link: https://chatgpt.com/g/g-p-6a3a407d2f30819197a4665bfa85dd41-metodologiia-soprovozhdeniia-proektirovaniia/c/6a4b827e-6cf4-83ea-be38-5cfe6de8f0aa?tab=sources

В терминах BDPF это не «отчёт по исходным данным» и не «чек-лист комплектности ИД». Это пакет извлечения проектно значимых сигналов:

```markdown
Исходные carriers
→ source signals
→ architectureStructuringNeed
→ unknown / candidate structures
→ вопросы к концепции использования, системе, архитектуре, ТУ, участку, доказательствам
```

BDPF прямо задаёт для такого случая вход BDPF.3: когда ТЗ, ИРД, ГПЗУ, изыскания, ТУ, требования эксплуатации, BIM/IFC, обследования или замечания переданы как материалы, но ещё не ясно, какой source signal из них извлечён, какую architectureStructuringNeed он создаёт, какую структуру затрагивает, где граница допустимого использования и когда нужен возврат к источнику.

## 1. Минимальный рабочий продукт

```markdown
InitialDataAnalysisPackage:
  packageId:
  projectRef:
  objectOrSiteRef:
  analysisStage: pre-design | concept | pre-PD | other
  analyzedByRoleAssignmentRefs:
  reviewedBySupportRoleAssignmentRefs:
  sourceCarrierRegisterRef:
  designSourceSignalUseRegisterRef:
  initialProblemToStructureFlowRef:
  sourceGapAndAmbiguityLogRef:
  sourceReturnAndChangeImpactRulesRef:
  readinessGateRef:
  nonAdmissibleUse:
```
%% нужна раскладка на русском языке %%

Фактически пакет состоит из 5 таблиц:
- Таблица 1 — реестр исходных carriers
- Таблица 2 — реестр DesignSourceSignalUseRecord
- Таблица 3 — матрица покрытия исходных данных
- Таблица 4 — первичная BuiltAssetProblemToStructureFlowCard
- Таблица 5 — журнал дефицитов и возврата к источнику

## 2. Таблица 1 — реестр исходных carriers

Это список того, что получено, но без притворства, что сам факт получения уже является анализом.

```markdown
SourceCarrierRegister:
  carrierId:
  sourceKind:
  titleOrName:
  sourceOwner:
  issueDateOrVersion:
  receivedDate:
  validityOrFreshnessBoundary:
  projectScope:
  relatedObjectOrSiteScope:
  status:
    available | partial | missing | obsolete | ambiguous | pending
  suspectedUse:
  nonAdmissibleUse:
  nextAction:
```
Типовые строки на старте:

| Source carrier                           | Что фиксировать                                                                   |
| ---------------------------------------- | --------------------------------------------------------------------------------- |
| ТЗ заказчика                             | Версия, владелец, границы проекта, заявленные цели, функции, режимы, ограничения. |
| Отраслевое / технологическое ТЗ          | Технология, эксплуатационные режимы, производственные потоки, спецтребования.     |
| Участок / ГПЗУ / ИРД                     | Границы, ограничения, сервитуты, ЗОУИТ, посадочные параметры, внешние интерфейсы. |
| ТУ / предварительные условия подключения | Мощности, точки подключения, ограничения, сроки, условия изменения.               |
| Изыскания / обследования, если есть      | Геология, существующие конструкции, сети, риски, неполнота данных.                |
| Нормативные и специальные требования     | Не как «норма выполнена», а как источник ограничений и проверочных claims.        |

Важное правило: carrier не равен source signal. Документ может быть получен, но проектно значимый сигнал из него ещё не извлечён.

## 3. Таблица 2 — реестр DesignSourceSignalUseRecord

Это основной артефакт анализа исходных данных.

Практическая форма:

| Поле                        | Что пишет проектировщик                                                                                                                                  |
| --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `sourceSignalRef`           | Уникальный ID сигнала, например `SS-TZ-001`.                                                                                                             |
| `sourceCarrierRef`          | Откуда извлечено: ТЗ, отраслевое ТЗ, ГПЗУ, ТУ и т.д.                                                                                                     |
| `extractedSignal`           | Не цитата целиком, а проектно значимое содержание: «требуется режим 24/7», «подключение 2 МВт ограничено точкой X», «участок имеет санитарный разрыв Y». |
| `needKind`                  | Какой тип проектной потребности создаёт сигнал.                                                                                                          |
| `affectedHolonOrScope`      | Весь ОКС, корпус, участок, инженерная система, технологическая зона, фасад, эвакуационный путь и т.д.                                                    |
| `supportedQuestion`         | Какой проектный вопрос этот сигнал поддерживает.                                                                                                         |
| `affectedStructureKindRefs` | Какие структуры затрагивает: пространственная, функциональная, инженерная, конструктивная, потоковая, интерфейсная.                                      |
| `admissibleUse`             | Для чего сигнал можно использовать сейчас.                                                                                                               |
| `nonAdmissibleUse`          | Чего он сам по себе не доказывает и не выбирает.                                                                                                         |
| `missingOrAmbiguousData`    | Что неясно, противоречиво или отсутствует.                                                                                                               |
| `sourceReturnCondition`     | Когда надо вернуться к источнику: новая версия ТЗ, уточнение ТУ, изменение участка, противоречие, переход к PAD.                                         |
| `receivingPatternRef`       | Куда сигнал передаётся дальше: `BDPF.3A`, `BDPF.4`, `BDPF.5`, `BDPF.6`, `BDPF.8`, `BDPF.9`.                                                              |

BDPF задаёт минимальное правило: нельзя ссылаться на исходные данные как на основание решения, пока не отвечено, какой source signal извлечён, какую структуру ОКС он делает неизвестной, спорной или ограниченной, и какой следующий DPF/FPF-owner должен принять claim.

### Классификация сигналов по needKind

| `needKind`                     | Пример на старте                                                                   | Что затрагивает                                                        | Что не решает                                                          |
| ------------------------------ | ---------------------------------------------------------------------------------- | ---------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| `useScenarioNeed`              | ТЗ или отраслевое ТЗ задаёт режимы работы, пользователей, технологический процесс. | Функционально-планировочная, потоковая, эксплуатационная структура.    | Не является готовой концепцией использования и не выбирает планировку. |
| `sitePlacementNeed`            | Участок, ГПЗУ, красные линии, ЗОУИТ, подъезды, ограничения посадки.                | Пространственная, внешне-интерфейсная, транспортная структура.         | Не выбирает объёмно-планировочное решение.                             |
| `regulatoryConstraintNeed`     | СП, ТР, СТУ, санитарные, пожарные, отраслевые ограничения.                         | Constraint, control, spatial, flow structures.                         | Не выбирает решение само по себе.                                      |
| `engineeringCapacityNeed`      | ТУ, мощности, точки подключения, трассы, лимиты.                                   | Engineering systems, interface, control structures.                    | Не доказывает достаточность инженерной схемы.                          |
| `geotechnicalOrStructuralNeed` | Изыскания, основания, нагрузки, существующие конструкции.                          | Несущая, материальная, конструктивная, construction method structures. | Не выбирает конструктивную схему без comparison.                       |
| `constructionRealizationNeed`  | Ограничения стройплощадки, этапность, доступность, временные состояния.            | Construction method, temporary, logistics structures.                  | Не заменяет work plan.                                                 |
| `informationRelianceNeed`      | BIM/IFC, обследование, реестр, цифровой двойник как источник.                      | Description/source relation, captured/lost structure.                  | Не делает модель ОКС, evidence, gate или decision.                     |

## 4. Таблица 3 — матрица покрытия исходных данных

Эта таблица отвечает на вопрос экспертизы: анализ был проведён или просто файлы просмотрели?

```markdown
SourceSignalCoverageMatrix:
  sourceSignalRef:
  sourceCarrierRef:
  needKind:
  affectedScope:
  affectedStructureKind:
  supportedProjectQuestion:
  receivingRecord:
  gapOrAmbiguity:
  sourceReturnCondition:
```
Пример:

| Сигнал                                                    | Источник                | `needKind`                     | Затронутая структура                                    | Следующий проектный вопрос                                          |
| --------------------------------------------------------- | ----------------------- | ------------------------------ | ------------------------------------------------------- | ------------------------------------------------------------------- |
| Требуется круглосуточная эксплуатация                     | ТЗ                      | `useScenarioNeed`              | FunctionalPlanningStructure / ControlOperationStructure | Какие режимы работы должны быть отражены в концепции использования? |
| Участок ограничен санитарным разрывом                     | ГПЗУ / ИРД              | `sitePlacementNeed`            | MaterialSpatialStructure / external interfaces          | Какие зоны участка становятся недоступны для размещения?            |
| Точка подключения водоснабжения только с северной стороны | ТУ                      | `engineeringCapacityNeed`      | EngineeringSystemsStructure / ModuleInterfaceStructure  | Как это влияет на трассировку и инженерные помещения?               |
| Нет подтверждённых геологических данных                   | Отсутствующее изыскание | `geotechnicalOrStructuralNeed` | StructuralLoadBearingStructure                          | Какие конструктивные решения пока нельзя принимать?                 |


## 5. Таблица 4 — первичная BuiltAssetProblemToStructureFlowCard
После извлечения сигналов проектировщик должен не выбирать решения, а показать переход:

```markdown
source signals
→ architectureStructuringNeed
→ unknown / candidate structures
→ next owner
```

Минимальная форма:

```markdown
InitialProblemToStructureFlow:
  flowId:
  describedHolonRef:
  boundedContextRef:
    holonScope:
    lifecycleSlice:
    nqdWorkState: SignalIntake | ProblemShaping | StructureKindTriage
    sourceMaturity:
    decisionMaturity:
  sourceSignalRefs:
  architectureStructuringNeed:
  unknownStructure:
  candidateStructureKindRefs:
  affectedArchitectureCharacteristics:
  currentStopOrReturnReason:
  neighboringOwnerForNextClaim:
```

## 6. Таблица 5 — журнал дефицитов и возврата к источнику

Это управляемый список того, что блокирует следующий проектный ход.

```markdown
SourceGapAndReturnLog:
  gapId:
  relatedSourceSignalRef:
  missingOrAmbiguousData:
  affectedStructureKind:
  blockedNextUse:
    useConcept | systemConcept | architectureQuestion | candidateSynthesis | comparison | PAD | evidence | documentationProjection
  requestedClarification:
  requestedFrom:
  deadlineOrWindow:
  interimAssumptionIfAllowed:
  nonAdmissibleUseUntilResolved:
  sourceReturnCondition:
```
Примеры:

| Дефицит                                         | Что блокирует                                                                      |
| ----------------------------------------------- | ---------------------------------------------------------------------------------- |
| Нет отраслевого ТЗ по технологическому процессу | Нельзя стабилизировать `UseConceptRecord` и функционально-планировочную структуру. |
| ТУ предварительные, без подтверждённой мощности | Нельзя принимать PAD по инженерной схеме без условия пересмотра.                   |
| Границы участка есть, но нет сервитутов / ЗОУИТ | Нельзя закрыть spatial/interface structure.                                        |
| Нет изысканий                                   | Нельзя считать конструктивную схему сравнимой по достаточным основаниям.           |

## 7. Что именно проверяет экспертиза на этом этапе

Экспертиза не должна заново делать анализ за проектировщика. Её проверка — это SourceAnalysisReadinessGate.

```markdown
SourceAnalysisReadinessGate:
  gateId:
  gateProfile: reviewReadiness | domainDecisionReadiness
  designStep: initialDataAnalysis
  objectScope:
  requiredRecords:
    - SourceCarrierRegister
    - DesignSourceSignalUseRegister
    - SourceSignalCoverageMatrix
    - InitialProblemToStructureFlow
    - SourceGapAndReturnLog
  gateDecision:
    pass | passWithReservations | hold | reject | abstain
  decisionBasis:
  blockingGaps:
  nextAllowedWork:
  nonAllowedWork:
  nonAdmissibleUse:
```
Gate нужен, когда требуется понять, можно ли переходить к следующему шагу проектирования или передавать материалы на экспертное рассмотрение

Ключевая проверка для исходных данных: данные должны быть не просто загружены, а привязаны к проектным вопросам. Gate при этом не выбирает проектное решение и не доказывает его допустимость; он только проверяет, собран ли достаточный проектный комплект для следующего шага работы или рассмотрения.

## 8. Сервисная цепочка с анализа исходных данных

| Шаг услуги                      | Делает проектировщик                                                                                 | Делает сопровождение / экспертиза                                                            | Артефакт                        |
| ------------------------------- | ---------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------- | ------------------------------- |
| 0.1. Приём исходных carriers    | Загружает ТЗ, отраслевое ТЗ, участок/ГПЗУ/ИРД, ТУ, изыскания, обследования, требования эксплуатации. | Проверяет, что carriers идентифицированы, имеют владельца, версию, дату, границу применения. | `SourceCarrierRegister`         |
| 0.2. Извлечение source signals  | Выделяет проектно значимые сигналы.                                                                  | Проверяет, что сигналы не заменены общими цитатами и привязаны к scope.                      | `DesignSourceSignalUseRegister` |
| 0.3. Классификация потребностей | Назначает `needKind`.                                                                                | Проверяет, что `needKind` не делает скачок к решению.                                        | `needKind` rows                 |
| 0.4. Привязка к структурам      | Показывает, какие структуры становятся неизвестными, спорными, ограниченными.                        | Проверяет наличие `affectedStructureKindRefs` и `unknownStructure`.                          | `SourceSignalCoverageMatrix`    |
| 0.5. Первичный P2S              | Показывает `architectureStructuringNeed` и следующий owner.                                          | Проверяет, что нет прямого перехода к PAD.                                                   | `InitialProblemToStructureFlow` |
| 0.6. Дефициты и source return   | Фиксирует отсутствующие / спорные данные и правила возврата.                                         | Проверяет, что дефициты блокируют конкретные следующие uses.                                 | `SourceGapAndReturnLog`         |
| 0.7. Gate завершения анализа    | Предъявляет пакет.                                                                                   | Ставит `pass / passWithReservations / hold / reject / abstain`.                              | `SourceAnalysisReadinessGate`   |

## 9. Практическая формулировка требования к проектировщику

```markdown
До начала разработки проектных решений проектировщик выполняет анализ исходных данных
в форме InitialDataAnalysisPackage.

Пакет должен показывать не только перечень полученных материалов,
а извлечённые из них source signals, тип созданной проектной потребности,
затронутый scope ОКС, affected structure kinds, допустимое использование сигнала,
недостающие или неоднозначные данные, source-return conditions
и следующий BDPF/FPF-owner, который принимает claim.

Экспертное сопровождение на этом этапе не выполняет анализ вместо проектировщика
и не принимает проектное решение. Оно проверяет наличие и минимальную достаточность
трассы:
source carrier → source signal → needKind → architectureStructuringNeed
→ unknown/candidate structure → next owner / stop reason.
```

## 10. Минимальный шаблон для старта проекта

```markdown
InitialDataAnalysisPackage@Project:
  packageId: IDA-[project]-001
  objectRef:
  siteRef:
  stage:
  analyzedBy:
  reviewBoundary:
  sourceCarrierRegister:
    - carrierId:
      sourceKind: customerTZ | industryTZ | site | GPZU | TU | survey | regulation | other
      sourceOwner:
      issueDateOrVersion:
      status:
      freshnessBoundary:
  designSourceSignalUseRecords:
    - sourceSignalRef:
      sourceCarrierRef:
      extractedSignal:
      needKind:
      affectedHolonOrScope:
      supportedQuestion:
      affectedStructureKindRefs:
      affectedArchitectureCharacteristics:
      admissibleUse:
      nonAdmissibleUse:
      missingOrAmbiguousData:
      sourceReturnCondition:
      receivingPatternRef:
  initialProblemToStructureFlow:
    describedHolonRef:
    boundedContextRef:
    nqdWorkState: SignalIntake | ProblemShaping | StructureKindTriage
    architectureStructuringNeed:
    unknownStructure:
    candidateStructureKindRefs:
    currentStopOrReturnReason:
    neighboringOwnerForNextClaim:
  sourceGapAndReturnLog:
    - gapId:
      relatedSourceSignalRef:
      blockedNextUse:
      requestedClarification:
      nonAdmissibleUseUntilResolved:
  readinessGate:
    gateProfile: reviewReadiness
    gateDecision:
    blockingGaps:
    nextAllowedWork:
    nonAllowedWork:
```
Итого: анализ исходных данных должен быть проведён не в виде пояснительного эссе, а в виде трассируемого source-signal пакета. Он показывает, что проектировщик понял, какие проектные вопросы и структуры возникают из ТЗ, отраслевого ТЗ, участка и ТУ, но ещё не подменяет этим проектное решение.
