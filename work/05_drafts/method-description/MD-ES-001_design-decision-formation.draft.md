# MD-ES-001 — MethodDescription draft: формирование проектного решения через концепцию использования и концепцию системы

**Статус:** `draft`
**Тип артефакта:** `MethodDescription`
**Слой:** `work/05_drafts`
**Источник P2W:** `work/01_problem-cards/PC-ES-002-managed-design-decision-formation.md`
**Контекст:** экспертное сопровождение проектирования объектов капитального строительства
**FPF-опора:** `A.3.2 U.MethodDescription`, `A.15 Role-Method-Work Alignment`, `E.10.D2`, `A.7 Strict Distinction`, `C.2.1 U.Episteme`, `A.10 Evidence Graph Referring`, `E.18.1 Principles-to-Work Transduction Path`

## 1. Назначение draft

Этот draft описывает минимальный способ формирования проектного решения через две промежуточные description epistemes:

- концепцию использования ОКС;
- концепцию системы.

Цель draft — задать проверяемую цепочку от исходных данных, потребностей и сценариев использования ОКС к концепциям, выбранному проектному решению, его evidence и публикации в ПД / ИК.

Документ удерживает P2W-выход из `PC-ES-002`:

> Проектное решение должно быть проверяемо связано с концепцией использования и концепцией системы до попадания в состав ПД / ИК.

Этот draft не является:

- стабилизированным методом в `pack/`;
- регламентом проектировщика;
- календарным WorkPlan;
- записью фактически выполненной Work;
- обязательным составом проектной документации;
- инструкцией SIGNAL DOCS;
- доказательством повышения качества проектных решений.

---

## 2. EntityOfConcern и граница применения

**Главный объект внимания:** объект капитального строительства как целевая проектируемая система.

**Непосредственный объект управления:** состояние формирования проектного решения об ОКС до его публикации в проектной документации / ИК.

**Методический объект этого draft:** способ предъявить, проверить и зафиксировать связь:

```text
ОКС / часть ОКС
→ исходные данные и ограничения
→ польза и сценарии использования
→ концепция использования
→ концепция системы
→ проектное решение
→ evidence
→ publication carrier: ПД / ИК / файл / модель / таблица
```

Граница применения: draft используется для одного проектного решения или группы тесно связанных проектных решений, когда нужно понять, можно ли предъявлять в ИК / ПД / экспертное рассмотрение исходные данные, концепции, варианты или выбранное решение как проверяемые материалы, связанные с формированием состояния проектного решения об ОКС.

За пределами draft:

- разработка проектных решений вместо проектировщика;
- принятие управленческих решений вместо заказчика;
- полная stage-gate модель проектирования;
- полный профиль `G0-G3`;
- platform-specific workflow статусов SIGNAL DOCS.

## 3. Базовые различения

| Объект | Роль в этом draft | Чем не является |
|---|---|---|
| ОКС / часть ОКС | целевая проектируемая система | не файл, не ИК, не комплект ПД |
| Исходные данные | description / evidence input для формирования и проверки решения | не проектное решение и не proof достаточности сами по себе |
| Проектное решение | утверждение о выбранной структуре, функции, параметре, связи, ограничении или способе реализации ОКС | не автоматически качественное решение |
| Концепция использования | description episteme о пользе, сценариях, пользователях, окружении и value-логике | не рабочий план и не доказательство качества |
| Концепция системы | description episteme о составе, функциях, границах и взаимодействиях системы | не сама система и не комплект ПД |
| Evidence | обоснования, расчеты, сравнения, traces, замечания, решения, протоколы или иные следы, поддерживающие claim | не наличие документа само по себе |
| ПД / ИК / файл | publication / carrier исходных данных, концепций, описания решения или evidence | не само решение и не proof по умолчанию |
| MethodDescription | описание способа работы | не WorkPlan и не выполненная Work |

## 4. Входы метода

Для запуска метода по конкретному проектному решению должен быть задан минимальный вход.

| Вход | Содержание | Минимальная проверка достаточности |
|---|---|---|
| `TargetSystemRef` | ОКС, часть ОКС, подсистема или аспект ОКС | можно назвать целевую систему, а не только раздел ПД / файл / ИК |
| `DecisionCandidateRef` | проектное решение или зона, где решение должно быть сформировано | понятно, о каком утверждении об ОКС идет речь |
| `SourceDataRefs` | исходные данные, требования, ограничения, результаты обследований, условия площадки | источник и применимость данных восстановимы |
| `NeedConstraintRefs` | потребности, ограничения, риски, требования пользователя / заказчика / нормативной среды | ясно, какую проблему или потребность должно закрыть решение |
| `UseScenarioRefs` | сценарии использования, эксплуатационные ситуации, пользовательские или технологические сценарии | сценарии связаны с ОКС или его частью |
| `ReviewMaterialRefs` | имеющиеся файлы, ИК, модели, таблицы, разделы ПД или иные материалы рассмотрения | carrier может содержать исходные данные, концепции, варианты, решения и evidence, но не подменяет их |
| `ReviewFrameRef` | контур будущего экспертного рассмотрения, если он уже задан | не смешан с gate-процедурой без decision rule |

Если вход не позволяет определить ОКС / часть ОКС, состав материалов рассмотрения не должен считаться достаточным для работы по проектному решению. Если проектное решение еще не сформировано, материалы могут быть приняты как ранняя поставка информации для рассмотрения исходных данных, потребностей, ограничений или концепций с явной фиксацией, что `SelectedDecisionRef` пока отсутствует.

В рамках услуги экспертному рассмотрению могут предъявляться не только выбранные проектные решения. В ИК или иной согласованный carrier может попадать информация начиная с исходных данных:

```text
исходные данные
→ потребности / ограничения
→ концепция использования
→ концепция системы
→ варианты
→ выбранное проектное решение
→ evidence
→ публикация в ПД
```

На ранних шагах эксперт оценивает достаточность, непротиворечивость и применимость исходных данных, а также проверяемость концепций. Это не означает, что исходные данные или концепции становятся проектным решением; они являются входами и промежуточными description epistemes, от качества которых зависит допустимость последующего решения.

## 5. Промежуточные epistemes

### 5.1. Концепция использования

Концепция использования фиксирует, зачем и в каких сценариях ОКС или его часть должны работать.

Минимальные поля:

| Поле | Содержание |
|---|---|
| `UsePurpose` | заявленная польза или функция для пользователя / заказчика / эксплуатации |
| `UseActors` | основные пользователи, операторы, обслуживающие роли или внешние участники |
| `UseScenarios` | ключевые сценарии использования, эксплуатации, обслуживания или взаимодействия |
| `UseConstraints` | ограничения, риски, условия среды, нормативные или технологические рамки |
| `ValueClaim` | какой полезный эффект должен поддержать проектируемый элемент ОКС |
| `OpenUseQuestions` | что пока неизвестно и влияет на выбор решения |

Acceptance probe для концепции использования:

> Можно ли объяснить, какую пользу или сценарий использования поддерживает проектное решение?

### 5.2. Концепция системы

Концепция системы фиксирует, какой системой должен стать ОКС / часть ОКС, чтобы поддержать концепцию использования.

Минимальные поля:

| Поле | Содержание |
|---|---|
| `SystemBoundary` | граница системы или подсистемы, к которой относится решение |
| `SystemFunctions` | функции, которые система должна выполнять |
| `SystemComposition` | ключевые элементы, подсистемы, связи или интерфейсы |
| `OperatingLogic` | как система работает в релевантных сценариях |
| `ExternalInterfaces` | связи с внешними системами, средой, пользователями или инфраструктурой |
| `SystemConstraints` | ограничения, влияющие на структуру и функционирование |
| `OpenSystemQuestions` | открытые вопросы по составу, границам или работе системы |

Acceptance probe для концепции системы:

> Можно ли показать, какую часть системы меняет или задает проектное решение и почему именно такая системная конфигурация поддерживает заявленное использование?

## 6. Последовательность действий

### Шаг 1. Идентифицировать целевую систему

Зафиксировать ОКС, часть ОКС, подсистему или аспект ОКС, к которому относится решение.

Выход шага:

- `TargetSystemRef`;
- граница рассмотрения;
- указание, что не является целевой системой: файл, ИК, раздел ПД, статус, замечание.

Критерий перехода:

> Целевая система названа так, что ее можно отличить от carrier и publication form.

### Шаг 2. Оценить исходные данные и ограничения

Проверить, достаточно ли исходных данных, требований, ограничений, условий площадки и исходных материалов для формирования концепции использования, концепции системы или проектного решения.

Выход шага:

- `SourceDataRefs`;
- `NeedConstraintRefs`;
- оценка достаточности, непротиворечивости и применимости исходных данных;
- открытые вопросы по исходным данным;
- связь исходных данных с ОКС / частью ОКС.

Критерий перехода:

> Можно понять, на каких исходных данных будет строиться концепция или решение, и какие ограничения проверки нужно явно удержать.

Если исходные данные переданы в ИК без концепций и без выбранного решения, это допустимая ранняя поставка информации для экспертного рассмотрения. Она не должна обозначаться как сформированное проектное решение.

### Шаг 3. Сформировать или уточнить концепцию использования

Связать проектируемый элемент ОКС с пользой, сценариями использования, пользователями, эксплуатационными условиями и ограничениями.

Выход шага:

- `UseConceptRef`;
- список сценариев;
- value / use claim;
- открытые вопросы использования.

Критерий перехода:

> Есть проверяемая причина, почему проектное решение вообще нужно и какую ситуацию использования оно должно поддержать.

### Шаг 4. Сформировать или уточнить концепцию системы

Показать, какой системной конфигурацией ОКС / часть ОКС должна поддержать концепцию использования.

Выход шага:

- `SystemConceptRef`;
- границы системы;
- функции;
- состав / связи;
- ограничения и интерфейсы.

Критерий перехода:

> Можно объяснить, какая системная структура или логика работы нужна для заявленного использования.

### Шаг 5. Сформировать варианты проектного решения

Задать один или несколько вариантов решения, которые реализуют концепцию системы.

Выход шага:

- `OptionSetRef`;
- описание вариантов;
- основания исключения вариантов, если вариантность не открывалась;
- риски и ограничения по каждому варианту.

Критерий перехода:

> Выбранное решение не выглядит произвольным: понятно, какие альтернативы рассматривались или почему альтернативы не открывались.

### Шаг 6. Выбрать проектное решение

Сформулировать выбранное решение как утверждение об ОКС:

```text
для такого ОКС / части ОКС выбирается такая структура / функция / параметр / связь / способ реализации, потому что...
```

Выход шага:

- `SelectedDecisionRef`;
- граница решения;
- связь с `UseConceptRef` и `SystemConceptRef`;
- основание выбора.

Критерий перехода:

> Решение можно отделить от текста, чертежа или модели, которыми оно будет описано.

### Шаг 7. Подготовить evidence

Собрать или указать evidence carriers, которые поддерживают claim о достаточности исходных данных, проверяемости концепции или допустимости решения.

Evidence подготавливается не "вообще к материалам", а к конкретному claim. Для каждого claim нужно указать, что именно подтверждается, каким carrier это подтверждается и какие ограничения проверки остаются.

Минимальные группы claims:

| Claim | Что подтверждается | Примеры evidence carriers |
|---|---|---|
| `SourceDataAdequacyClaim` | исходные данные достаточны, применимы или имеют явно зафиксированные ограничения | перечень исходных данных, результаты проверки полноты, ответы на запросы, протоколы, замечания по недостающим данным |
| `UseConceptTraceClaim` | концепция использования связана с потребностями, сценариями и ограничениями ОКС | описание сценариев, пользовательские / эксплуатационные требования, decision note, экспертные вопросы и ответы |
| `SystemConceptTraceClaim` | концепция системы выводима из концепции использования и ограничений | схема состава, функциональная схема, модель взаимодействий, матрица функций, экспертное заключение по концепции |
| `OptionRationaleClaim` | варианты рассмотрены или причина отсутствия вариантности явно зафиксирована | таблица сравнения вариантов, расчет, протокол выбора, запись об ограничении вариантности |
| `SelectedDecisionClaim` | выбранное проектное решение сформулировано и обосновано | расчет, обоснование выбора, журнал принятых проектных решений, decision note |
| `PublicationTraceClaim` | материалы опубликованы в carrier и сохраняют связь с рассматриваемыми claims | версия ИК, файл, раздел ПД, модель, таблица, ссылка на состав материалов |

Допустимые типы evidence carriers:

- исходные данные и требования;
- результаты оценки исходных данных;
- рабочие записи по концепции использования и концепции системы;
- расчеты;
- сравнение вариантов;
- протокол или decision note;
- экспертные вопросы и ответы;
- замечания и их закрытие;
- ссылки на модели, схемы, таблицы;
- журнал принятых проектных решений;
- отчет или фрагмент evidence trail.

Выход шага:

- `EvidenceClaimMap`;
- `EvidenceRefs`;
- связь каждого evidence carrier с конкретным claim;
- ограничения evidence: что carrier подтверждает, чего не подтверждает, где требуется дополнительная проверка;
- missing evidence: какие claims пока не имеют достаточного подтверждения.

Критерий перехода:

> Есть не просто carrier, а восстановимая связь: claim об исходных данных, концепции или решении -> evidence carrier -> источник / автор / дата / версия / ограничение применимости.

### Шаг 8. Опубликовать материалы в carrier

Отразить исходные данные, концепции, варианты, выбранное решение и evidence в ПД / ИК / файле / модели / таблице или другом carrier.

Выход шага:

- `PublicationRefs`;
- связь publication с `SourceDataRefs`, `UseConceptRef`, `SystemConceptRef`, `OptionSetRef` или `SelectedDecisionRef`;
- версия carrier;
- ограничения публикации.

Критерий перехода:

> Публикация показывает, где материалы описаны, но не подменяет исходные данные, концепции, решение и evidence.

### Шаг 9. Предъявить материалы на экспертное рассмотрение

Предъявление материалов допустимо, если можно восстановить применимую трассу. Для ранней поставки информации она может остановиться на исходных данных или концепциях:

```text
TargetSystemRef
→ SourceDataRefs
→ UseConceptRef
→ SystemConceptRef
→ SelectedDecisionRef
→ EvidenceRefs
→ PublicationRefs
```

Выход шага:

- Либо исходные данные готовы к оценке;
- либо концепция использования готова к оценке;
- либо концепция системы готова к оценке;
- либо варианты проектного решения готовы к оценке;
- либо выбранное решение готово к экспертному рассмотрению;
- либо фиксируется недостаточность: чего не хватает для рассмотрения;
- либо фиксируется ограничение проверки.

Критерий перехода:

> Эксперт может проверять состояние формирования проектного решения начиная с исходных данных и концепций, а не только движение документа / ИК или уже выбранное решение.

## 7. Роли

Роли в этом draft описывают ответственность внутри метода. Они не являются штатной структурой и не требуют отдельной должности для каждой роли. Один участник может совмещать несколько ролей, если сохраняется различение ответственности.

| Роль | Ответственность в методе | Не делает |
|---|---|---|
| Владелец исходных данных / ограничений | отвечает за происхождение, актуальность, полноту, версию и область применимости исходных данных; фиксирует ограничения, допущения и неизвестные данные; отвечает на запросы по исходным данным; указывает, какие данные нельзя использовать без дополнительной проверки | не превращает наличие исходных данных в evidence достаточности и не принимает проектное решение |
| Заказчик / представитель потребителя | подтверждает потребности, сценарии использования, эксплуатационные ограничения и допустимость value / use claim; помогает различить бизнес-потребность, пользовательский сценарий и техническое решение; фиксирует изменения потребностей, влияющие на концепцию использования | не заменяет техническое обоснование решения и не закрывает инженерные риски административным согласием |
| Руководитель проекта со стороны проектировщика | обеспечивает согласованность работы проектной команды; назначает ответственных за исходные данные, концепции, варианты, решение и ответы на замечания; контролирует полноту предъявления материалов; снимает межраздельные противоречия; обеспечивает единую позицию проектировщика по `UseConceptRef`, `SystemConceptRef`, `OptionSetRef` и `SelectedDecisionRef` | не является автором всех решений, не заменяет экспертов по направлениям и не является evidence качества решения сам по себе |
| Проектировщик / разработчик решения | формирует или актуализирует концепцию использования, концепцию системы, варианты и выбранное проектное решение; связывает решение с исходными данными и ограничениями; готовит описания, расчеты, схемы, модели, rationale и другие evidence carriers; отвечает на замечания по своему решению | не перекладывает выбор решения на эксперта и не считает публикацию в carrier доказательством допустимости решения |
| Эксперт по направлению | оценивает исходные данные, концепцию использования, концепцию системы, варианты и проектное решение в границах своего направления; формирует замечания, вопросы, риски и evidence claims; указывает недостающие данные и ограничения проверки; проверяет, не противоречит ли решение предметным критериям направления | не проектирует вместо проектировщика, не принимает междисциплинарное решение единолично и не подменяет decision owner |
| Ведущий эксперт | задает рамку предъявления, трассировки и проверяемости исходных данных, концепций, вариантов и решения; координирует экспертов по направлениям; проверяет восстановимость `EvidenceClaimMap`; фиксирует ограничения рассмотрения; обеспечивает, чтобы carrier / status не подменяли состояние проектного решения | не проектирует вместо проектировщика и не заменяет экспертов по направлениям в предметной оценке |
| Ответственный за carrier / ИК | обеспечивает публикацию, версионирование и восстановимость материалов; фиксирует состав материалов, версию ИК, файлы, связи с `SourceDataRefs`, `UseConceptRef`, `SystemConceptRef`, `OptionSetRef`, `SelectedDecisionRef` и `EvidenceRefs`; сохраняет историю изменений и доступность ссылок | не объявляет статус carrier evidence качества решения и не меняет смысл материалов при публикации |
| Принимающая сторона / decision owner | принимает или подтверждает проектное решение в пределах своих полномочий; фиксирует принятое решение, условия принятия, ограничения и открытые вопросы; использует evidence и результаты экспертного рассмотрения при принятии | не заменяет evidence административным статусом и не снимает ответственность авторов решения за rationale |

Если роль не назначена, это не блокирует draft как описание метода, но блокирует применение метода к конкретному проектному решению или поставке информации без фиксации ответственности.

## 8. Выход метода

Минимальный выход по одному проектному решению или ранней поставке информации для формирования решения:

| Выход | Содержание |
|---|---|
| `SourceDataAssessmentRef` | оценка достаточности, непротиворечивости и применимости исходных данных |
| `UseConceptAssessmentRef` | оценка восстановимости концепции использования и сценариев |
| `SystemConceptAssessmentRef` | оценка восстановимости концепции системы, границ, функций и состава |
| `DecisionStateRecord` | состояние проектного решения: сформировано / вариантно / выбрано / обосновано / опубликовано / готово к рассмотрению / принято с ограничениями |
| `SourceDataRefs` | исходные данные, требования и ограничения, на которых строятся концепции и решение |
| `UseConceptRef` | ссылка на концепцию использования или ее рабочую запись |
| `SystemConceptRef` | ссылка на концепцию системы или ее рабочую запись |
| `SelectedDecisionRef` | формулировка выбранного проектного решения, если оно уже сформировано |
| `EvidenceRefs` | evidence carriers, поддерживающие оценку исходных данных, концепций или выбранного решения |
| `PublicationRefs` | где материалы отражены в ПД / ИК / файле / модели / таблице |
| `OpenIssues` | открытые вопросы, замечания, ограничения проверки |

Минимальная текстовая форма выхода:

```text
Для [ОКС / части ОКС] переданы исходные данные [source data refs].
Исходные данные оценены как [source data assessment].
В сценарии [use scenario] требуется [value / function].
Концепция использования оценена как [use concept assessment].
Система должна работать как [system concept].
Концепция системы оценена как [system concept assessment].
Если решение уже выбрано: выбрано проектное решение [selected decision], потому что [rationale].
Допустимость исходных данных / концепций / решения подтверждается [evidence refs].
Материалы опубликованы в [publication refs].
Остаются открытыми [open issues / limitations].
```

## 9. Критерии допустимости передачи в ПД / ИК / экспертное рассмотрение

Материалы по формированию проектного решения допустимо передавать в ПД / ИК / экспертное рассмотрение, если выполнены минимальные критерии:

| Критерий | Проверочный вопрос |
|---|---|
| Целевая система определена | Ясно, к какому ОКС / части ОКС относится решение? |
| Исходные данные оценимы | Понятно, какие исходные данные переданы и какие ограничения их применимости есть? |
| Концепция использования восстановима | Понятно, какую пользу, сценарий или потребность поддерживает решение? |
| Концепция системы восстановима | Понятно, какая системная конфигурация требуется? |
| Стадия рассмотрения названа | Понятно, рассматриваются исходные данные, концепция, варианты или выбранное решение? |
| Решение сформулировано как claim об ОКС, если оно уже выбрано | Можно отделить решение от документа, файла или модели? |
| Основание выбора зафиксировано, если выбор уже сделан | Понятно, почему выбран этот вариант? |
| Evidence связано с claim | Есть evidence carriers, поддерживающие именно исходные данные, концепцию или решение? |
| Publication не подменяет evidence | Статус carrier не используется как доказательство качества решения сам по себе? |
| Открытые вопросы видны | Ограничения, риски и замечания не потеряны? |

Если один или несколько критериев не выполнены, материалы могут быть опубликованы как ранняя или черновая поставка информации, но не должны считаться сформированным, обоснованным и готовым проектным решением без фиксации ограничения.

## 10. Failure modes

| Failure mode | Симптом | Коррекция |
|---|---|---|
| `FM-01 Publication-as-decision` | решение считается готовым, потому что оно появилось в ПД / ИК | отделить `SelectedDecisionRef` от `PublicationRefs` |
| `FM-02 Use concept skipped` | решение не связано с пользой и сценариями использования | вернуть шаг 2 и зафиксировать `UseConceptRef` |
| `FM-03 System concept skipped` | решение не связано с составом, функцией и границей системы | вернуть шаг 3 и зафиксировать `SystemConceptRef` |
| `FM-04 Evidence-by-status` | статус файла / ИК используется как доказательство качества решения | заменить статус на evidence graph: claim -> carrier -> source / version |
| `FM-05 Expert as designer` | эксперт фактически выбирает решение вместо проектировщика | вернуть ответственность проектировщику; эксперт задает критерии предъявления и проверяемости |
| `FM-06 WorkPlan-as-method` | календарь или список задач выдается за метод | выделить MethodDescription отдельно от WorkPlan |
| `FM-07 MethodDescription-as-work` | наличие этого draft считается выполненной работой по решению | фиксировать конкретную Work-запись отдельно |
| `FM-08 Decision-only review` | услуга начинает рассматривать только выбранное решение и теряет оценку исходных данных и концепций | разрешить раннюю поставку информации в ИК / ином carrier от исходных данных до концепций и вариантов |

## 13. Следующий P2W-use

> Провести пилот на одной поставке информации / версии ИК, которая начинается с исходных данных, и заполнить acceptance probe из раздела 11.

Минимальный результат пилота:

- заполненная таблица по одной поставке информации / версии ИК;
- список missing fields;
- список мест, где carrier / status начинает подменять исходные данные, концепции, решение или evidence;

До пилота этот draft остается рабочей MethodDescription-гипотезой и не переносится в `pack/`.
