## 1. Граница пилота

Пилот применяет `UseConceptRecord` к ранней поставке информации, где видимы только исходные данные и идентификационные признаки ОКС. Запись не переносится в `pack/`, не создает стабильный шаблон и не доказывает наличие концепции использования.

**Entity-of-concern:** один обезличенный объект капитального строительства как целевая проектируемая система, представленный в ранней версии ИК `G0-ИРД`.

**Carrier пилота:** версия ИК / поставка информации `G0-ИРД / Идентификация ОКС`, основной файл `FILE-G0-IRD-01` — форма идентификации объекта.

**Ограничение факта:** заполненный экземпляр ИК с именем объекта, проектом, датой и фактическими строками исходных данных в текущем репозитории не найден. Поэтому пилот фиксирует рабочий срез `IK-G0-IRD-v0-source-identification-only` как обезличенный carrier, а не как полную evidence-запись по реальному проекту.

**Классификация изменения:** `work`, пилотная проверка рабочей записи. Не `pack`, не platform-specific инструкция.

## 2. UseConceptRecord

### TargetSystemRef:

`OKS-ANON-G0-IRD-01`

ОКС целиком на уровне первичной идентификации. Фактическое название объекта, адрес, проект / договор и версия / дата ИК в доступном carrier не восстановлены.

### SourceDataRefs:

- `FILE-G0-IRD-01` — форма идентификации объекта: идентификационные признаки объекта, базовые параметры, источники данных.
- `FILE-G0-IRD-02` — пояснение по спорным или вариантным параметрам, если такие параметры есть; в пилотном срезе наличие не подтверждено.
- `Выписка по объекту / титульные сведения` — обязательное приложение, подтверждающее базовую идентификацию кейса; фактический файл в пилотном срезе не предъявлен.

### NeedConstraintRefs:

- `NC-G0-IRD-01` — поля идентификации должны быть однозначно читаемы.
- `NC-G0-IRD-02` — критичные признаки должны иметь источники данных.
- `NC-G0-IRD-03` — спорные или вариантные параметры требуют короткого пояснения без дублирования таблицы.
- `NC-G0-IRD-04` — риск неполноты исходных сведений должен быть явно виден как ограничение дальнейшей работы.

Эти refs описывают ограничения применимости исходных данных и carrier. Они не являются потребностями пользователей ОКС и не заменяют концепцию использования.

### UsePurpose:

Не подтверждено для ОКС.

Из carrier восстанавливается только рабочая цель поставки: едино идентифицировать кейс для последующего рассмотрения. Это цель carrier / поставки информации, а не заявленная польза или функция проектируемого ОКС.

### UseActors:

Не подтверждены как actors использования ОКС.

Из доступного carrier можно восстановить только участников подготовки / проверки информации: проектная команда, ответственный за подготовку ИК, сторона экспертизы. Эти роли относятся к работе с carrier и не считаются пользователями, операторами или эксплуатационными ролями целевой системы.

### UseScenarios:

Не подтверждены.

Поставка `G0-ИРД` не содержит сценарии использования, эксплуатации, обслуживания или взаимодействия с ОКС. Единственный восстановимый сценарий относится к workflow: передать идентификационные данные в Signal DOCS / ИК для последующего рассмотрения. Это не use scenario ОКС.

### UseConstraints:

Не подтверждены как ограничения использования ОКС.

В пилотном carrier видны только ограничения данных:

- читаемость идентификационных полей;
- наличие источников у критичных признаков;
- необходимость пояснения спорных / вариантных параметров;
- риск неполноты исходных сведений.

Эти ограничения помогают оценить пригодность исходных данных, но не описывают эксплуатационные, пользовательские, технологические или нормативные ограничения использования ОКС.

### ValueClaim:

Live value claim об ОКС отсутствует.

Кандидатный claim, допустимый только как рабочая гипотеза: корректная идентификация ОКС позволит не смешивать разные объекты / части ОКС в последующих поставках и экспертном рассмотрении. Этот claim требует evidence по фактическому объекту и не является концепцией использования.

### OpenUseQuestions:

- Какую пользу должен обеспечивать ОКС для заказчика / пользователей / эксплуатации?
- Какие основные группы пользователей, операторов, обслуживающих ролей или внешних участников связаны с ОКС?
- Какие сценарии использования или эксплуатации являются определяющими для дальнейших проектных решений?
- Какие ограничения использования ОКС уже заданы исходными данными, заданием на проектирование, градостроительной, технологической или нормативной рамкой?
- Достаточны ли идентификационные признаки для выделения именно этого ОКС, а не только карточки / ИК / набора файлов?
- Есть ли часть ОКС или подсистема, для которой concept-of-use должен быть выделен отдельно?

### UseConceptTraceClaim:

`UseConceptTraceClaim-G0-IRD-01`: по доступной поставке концепция использования не является live description episteme.

Трасса не доходит до `UsePurpose`, `UseActors`, `UseScenarios`, `UseConstraints` и `ValueClaim` как проверяемых claims об ОКС.

### MissingEvidence:

- Фактическое имя / шифр ОКС.
- Версия / дата конкретной поставки ИК.
- Фактически заполненный `FILE-G0-IRD-01`.
- Фактические источники данных по критичным идентификационным признакам.
- Приложение с выпиской по объекту / титульными сведениями.
- Пояснение по спорным или вариантным параметрам, если такие параметры есть.
- Задание на проектирование или иной source, где заявлена польза ОКС.
- Описание пользователей / операторов / эксплуатационных ролей.
- Описание сценариев использования / эксплуатации / обслуживания.
- Ограничения использования ОКС, отличные от ограничений качества исходных данных.
- Evidence, связывающее use claims с исходными данными и границей ОКС.

### PublicationRefs:

- `IK-G0-IRD-v0-source-identification-only` — рабочее имя пилотного среза; не внешний стабильный ID.
- `G0-ИРД / Идентификация ОКС`.
- `FILE-G0-IRD-01 / Форма идентификации объекта`.
- `docs/forms/01-object-identification-template.xlsx` — указан в реестре как template для основного файла ИК, но фактический заполненный экземпляр не найден.

Publication refs фиксируют, где материал должен быть опубликован или представлен. Они не являются evidence концепции использования и не доказывают состояние проектного решения.

### UseConceptAssessment:

`not-live / insufficient-for-use-concept`

По ранней поставке `G0-ИРД` можно удержать целевой ОКС как объект внимания и начать оценку исходных данных. Концепция использования ОКС не восстановима: нет подтвержденных purpose, actors, scenarios, use constraints и value claim. Материал допустим как ранняя поставка исходных данных / идентификационных признаков, но не должен считаться сформированной концепцией использования или проектным решением.

## 3. Acceptance probe по пилоту

| Проверочный вопрос | Ответ по пилоту |
|---|---|
| Можно назвать ОКС / часть ОКС? | Частично: выбран обезличенный ОКС, но фактический идентификатор объекта отсутствует. |
| Можно отделить ОКС от carrier? | Да: `G0-ИРД` и `FILE-G0-IRD-01` являются carrier, не целевой системой. |
| Есть исходные данные? | Частично: известен ожидаемый тип исходных данных, но фактические строки и sources не предъявлены. |
| Есть потребности / ограничения использования? | Нет. Есть только ограничения качества исходных данных. |
| Есть use purpose / actors / scenarios? | Нет. |
| Есть value claim об ОКС? | Нет, только кандидатная гипотеза о ценности корректной идентификации. |
| Есть evidence для use concept? | Нет. Evidence поддерживает только вывод о недостаточности для use concept. |
| Publication подменяет evidence? | Нет, если сохранять текущую оценку `not-live` и не объявлять статус ИК доказательством use concept. |

## 4. Вывод для следующего шага

`UseConceptRecord` нужен как отдельная рабочая запись хотя бы для ранних поставок: он показывает, где carrier уже позволяет удержать `TargetSystemRef` и `SourceDataRefs`, но еще не позволяет заявлять концепцию использования.

Следующий минимальный запрос к поставке:

1. заполненный экземпляр `FILE-G0-IRD-01` с версией / датой;
2. источники по критичным идентификационным признакам;
3. краткая запись `UsePurpose`, `UseActors`, `UseScenarios`, `UseConstraints` от заказчика / проектной команды;
4. явное решение, относится ли use concept ко всему ОКС или к части / подсистеме.
